# Flequit Claude Code Skills

このディレクトリには、Flequit プロジェクト専用の Claude Code skills が含まれています。

## スキル一覧

### 1. フロントエンドテスト (`frontend-testing`)

**用途**: フロントエンド（TypeScript/Svelte 5/Vitest）のテスト実装とデバッグ

**主な機能**:
- Vitest テストの作成・実行
- Svelte 5 コンポーネントのテスト
- 国際化（i18n）のモック設定
- テストエラーの修正

**使用例**:
- "Vitest でこのユーティリティ関数のテストを書いて"
- "このコンポーネントのテストが失敗している。修正して"
- "翻訳を含むテストを作成して"

### 2. バックエンドテスト (`backend-testing`)

**用途**: バックエンド（Rust/Tauri）のテスト実装とデバッグ

**主な機能**:
- cargo test の実行（必ず `-j 4` 指定）
- Repository/Service/Facade レイヤーのテスト
- Mock を使用した単体テスト
- トランザクションのテスト

**使用例**:
- "このリポジトリのテストを書いて"
- "cargo test が失敗している。修正して"
- "トランザクションを含むテストを作成して"

### 3. Tauri コマンド実装 (`tauri-command`)

**用途**: Tauri コマンドの実装とフロントエンド・バックエンド間の通信

**主な機能**:
- camelCase ⇔ snake_case の自動変換対応
- 統一されたエラーハンドリング
- 戻り値の統一パターン
- 型安全な通信実装

**使用例**:
- "新しい Tauri コマンドを追加して"
- "フロントエンドから Tauri コマンドを呼び出すコードを書いて"
- "Tauri コマンドのパラメータ名が間違っている。修正して"

### 4. アーキテクチャ準拠チェック (`architecture-review`)

**用途**: クリーンアーキテクチャ、レイヤードアーキテクチャへの準拠チェック

**主な機能**:
- Crate 間のアクセス制御チェック
- レイヤー間の依存関係チェック
- Store と Service の分離チェック
- アンチパターンの検出

**使用例**:
- "このコードがアーキテクチャに準拠しているかチェックして"
- "レイヤー違反がないか確認して"
- "Store が直接 invoke を呼んでいないかチェックして"

### 5. デバッグ支援 (`debugging`)

**用途**: フロントエンドとバックエンドのデバッグ

**主な機能**:
- 型エラーの修正
- 実行時エラーの解析
- Tauri 通信のデバッグ
- ログの確認とトラブルシューティング

**使用例**:
- "この型エラーを修正して"
- "なぜこのコードが動かないのか調べて"
- "Tauri コマンドが呼ばれていない。デバッグして"

### 6. 国際化対応 (`i18n`)

**用途**: Inlang Paraglide を使用した国際化（i18n）対応

**主な機能**:
- 翻訳メッセージの追加
- reactiveMessage の使用
- パラメータ付きメッセージ
- テストでの翻訳モック

**使用例**:
- "この UI テキストを国際化対応にして"
- "新しい翻訳キーを追加して"
- "言語切り替え機能を実装して"

### 7. ドキュメント編集 (`documentation`)

**用途**: プロジェクトドキュメントの作成・編集（日本語・英語の両方）

**主な機能**:
- ドキュメントの追加・更新
- 日本語版と英語版の同時更新（必須）
- Markdown フォーマット
- ドキュメント構造の維持

**使用例**:
- "新しい設計ドキュメントを作成して"
- "このドキュメントを更新して（日英両方）"
- "ドキュメントの翻訳が古い。更新して"

**🚨 重要**: ドキュメントを編集する際は、**必ず日本語版（`docs/ja/`）と英語版（`docs/en/`）の両方を同時に更新**してください。

### 8. コーディング標準 (`coding-standards`)

**用途**: Flequit プロジェクトのコーディング標準への準拠チェック

**主な機能**:
- 命名規則のチェック
- ファイル構造のチェック
- 型定義のチェック
- インポート順序のチェック

**使用例**:
- "このコードがコーディング標準に準拠しているかチェックして"
- "命名規則に違反していないか確認して"
- "ファイルが200行を超えているので分割して"

## スキルの自動実行

Claude Code は、ユーザーのリクエストに基づいて自動的に適切なスキルを選択・実行します。

例:
- "テストを書いて" → `frontend-testing` または `backend-testing` が自動的に選択
- "Tauri コマンドを追加して" → `tauri-command` が自動的に選択
- "国際化対応にして" → `i18n` が自動的に選択

## スキルの特徴

### Progressive Disclosure

各スキルは、本質的な情報を SKILL.md に、詳細資料は別ファイルに分割する Progressive Disclosure パターンを採用しています。必要に応じて `docs/` 配下のドキュメントを参照してください。

### プロジェクト固有の知識

これらのスキルは、Flequit プロジェクトの以下の特徴に特化しています：

- **Tauri + SvelteKit** アーキテクチャ
- **Svelte 5 runes** (`$state`, `$derived`, `$effect`)
- **Inlang Paraglide** による国際化
- **クリーンアーキテクチャ**（Crate 分離）
- **camelCase ⇔ snake_case** 自動変換
- **プロジェクト固有のテスト戦略**

### ツール制限

セキュリティとパフォーマンスのため、各スキルには `allowed-tools` が設定されています。これにより、スキルが適切なツールのみを使用することを保証しています。

## トラブルシューティング

### スキルが自動実行されない

**原因**: description が曖昧、またはキーワードが不足している

**解決**: より具体的なリクエストを行う
- ❌ "コードを書いて"
- ✅ "Vitest でこの関数のテストを書いて"

### YAML 構文エラー

**確認事項**:
- ファイル1行目が `---` であること（空行なし）
- インデントがスペース（タブではない）
- 終了の `---` があること

### スキルが見つからない

スキルは `.claude/skills/` ディレクトリに配置されている必要があります。

```
.claude/skills/
├── skill-name/
│   └── SKILL.md
```
