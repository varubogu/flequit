# テスト要件

実装ルール (テストフレームワーク、配置、コマンド、設計原則) は [`../rules/testing.md`](../rules/testing.md) を参照。本書は **品質要件としてのテスト戦略** を定義する。

## CI/CD

- GitHub Actions
  - PR 時に自動テスト実行 (Lint + 単体 + 統合)
  - マージ時に自動デプロイ
  - 日次で E2E テスト

## テストデータ環境

- ローカル開発: SQLite
- CI: SQLite
- ステージング: 本番と同等

## テスト種別と範囲

- **単体テスト**: ビジネスロジック、データアクセス層、UI コンポーネント、ユーティリティ
- **統合テスト**: API エンドポイント、DB 結合、同期処理、認証/認可、多言語対応 (キーと翻訳の整合)
- **E2E**: マルチアカウント (10 アカウント同時/同期/競合解決)、タスク管理基本、オフライン対応、多言語対応
- **パフォーマンス**: 同時接続 10 アカウント、レスポンスタイム (ローカル DB 1s 以内 / サーバ 5s 以内 / 全体同期 10 分以内)、負荷 (通常 5 / 最大 10 / 過負荷 15)

## 品質基準

### コードカバレッジ

- ビジネスロジック: 90% 以上
- UI コンポーネント: 80% 以上
- 統合テスト: 70% 以上
- E2E テスト: 主要フロー 100%

### パフォーマンス

- レスポンスタイム基準値の遵守
- メモリリーク: 検出されないこと
- クラッシュ: 発生しないこと

### 品質ゲート (PR マージ条件)

- 全テスト成功
- カバレッジ基準達成
- Lint エラーなし
- コードレビュー承認

## テストデータセット

- 最小 (基本機能確認)
- 標準 (一般的な使用パターン)
- 大規模 (パフォーマンステスト用): タスク 10,000 件、プロジェクト 100 件、タグ 1,000 件

## 障害対応

- テスト失敗時: 原因分析 → 再現手順文書化 → 修正 → 再テスト → CI 一時停止判断
- 本番障害防止: カナリアリリース、フィーチャーフラグ、ロールバック手順、監視体制
