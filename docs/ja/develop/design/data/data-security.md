# データセキュリティ設計書

Flequit における包括的なデータセキュリティ設計。データ盗取に対する防御を提供し、実運用に耐える堅牢性を実現する。

## 1. ID 設計とセキュリティ

### 二重 ID 設計

セキュリティ強化のため、アカウント・ユーザー情報は二重 ID 構造を採用する。

| 種別 | 用途 | セキュリティレベル | 使用場面 |
| --- | --- | --- | --- |
| **内部 ID (`AccountId`)** | システム内部識別 | 機密 | DB 主キー、内部処理のみ |
| **公開 ID (`UserId`)** | 外部参照・共有 | 公開可能 | プロジェクト参加、タスク割り当て |

### 原則

1. **内部 ID 保護**: システム外への漏洩を絶対禁止
2. **公開 ID 利用**: プロジェクト共有・招待は公開 ID のみ
3. **ID 分離**: 内部処理と外部連携で使用する ID を明確分離
4. **漏洩対策**: 内部 ID 漏洩時も公開 ID からの逆引きを困難化

### 実装ガイドライン

- API レスポンスに内部 ID を含めない (`Account.user_id` のみ返却)
- ログ出力時は公開 ID のみ記録
- エラーメッセージに内部 ID を表示しない
- フロントエンドは公開 ID (`Account.user_id`, `User.id`) のみ扱う
- DB 参照: Account は `user_id`、User は `id`

エンティティ定義は `entity/accounts-and-users.md` を参照。

## 2. データ構造とファイル管理

### Automerge ドキュメント単位

- **アカウント情報**: アカウントごとに 1 ファイル (`local-account.am`, `<account-id>.am`)
- **プロジェクト情報**: プロジェクトごとに 1 ファイル (`<project-id>.am`)。プロジェクト〜サブタスク、協力者情報を含む

### ローカル保管先

| OS | パス |
| --- | --- |
| Windows | `C:\Users\<username>\AppData\Local\Flequit` |
| macOS | `~/Library/Application Support/Flequit` |
| Linux | `$XDG_DATA_HOME/flequit` → `~/.local/share/flequit` → `~/.flequit` (フォールバック順) |

### フォルダ構造 (例)

```
Flequit/
├── account/
│   ├── local-account.am     # ローカルアカウント (常に 1 つ)
│   ├── <account-id1>.am     # ログインアカウント
│   └── <account-id2>.am
└── projects/
    ├── <project-id1>.am
    └── <project-id2>.am
```

### クラウドストレージ・Web サーバー

アカウント情報に該当ファイルへのパスを保持する。

- **クラウドストレージ**: 任意フォルダを設定
- **Web バックエンド**: Amazon S3 等のオブジェクトストレージを想定

## 3. 暗号化システム

- **アルゴリズム**: AES-256
- **保存形式**: 全 Automerge ドキュメントを暗号化状態で保存
- **実行時**: RAM 上でのみ復号化して Automerge 操作を実行
- **再暗号化**: 操作後に再暗号化してから保存
- **鍵管理**: 階層的鍵構造による細やかなアクセス制御

## 4. アクセス制御

### 基本方針

1. **ローカルアクセス**: 当該デバイス所有者のみ
2. **プロジェクト単位制御**: プロジェクトごとに独立した権限管理
3. **招待制**: 明示的な招待による鍵発行が必要
4. **即座失効**: 任意ユーザーを任意時点で失効可能
5. **最小権限原則**

### 権限階層

```
ローカルアカウント (デバイス所有者)
└── ログインアカウント (サービスアカウント)
    └── プロジェクト (所有者)
        └── ユーザー (編集 / 閲覧 / コメント)
```

### 権限レベル

- **所有者**: 完全制御 (削除、メンバー管理、権限変更)
- **編集者**: データ作成・編集・削除
- **閲覧者**: 読み取りのみ
- **コメンター**: 閲覧 + コメント投稿

## 5. 鍵管理

### 鍵階層

1. **マスター鍵**: ローカルアカウントが保持する最上位鍵
2. **アカウント鍵**: マスター鍵で暗号化
3. **プロジェクト鍵**: アカウント鍵で暗号化
4. **ユーザー専用鍵**: プロジェクト鍵をユーザー公開鍵で暗号化

### 招待・鍵発行フロー

1. 所有者が招待 (メール + 権限レベル)
2. 招待トークン生成 (有効期限付き)
3. 被招待者が承諾 + 公開鍵送信
4. プロジェクト鍵をユーザー公開鍵で暗号化してストレージに保存
5. 被招待者にアクセス権付与完了通知

### 鍵ローテーション

- **定期**: プロジェクト鍵を 6 ヶ月ごとに更新
- **侵害時**: 即座にプロジェクト鍵更新 + 全ユーザー再配布
- **メンバー変更時**: 追加・削除時にプロジェクト鍵更新

## 6. ユーザー失効

### 即座失効の仕組み (多重防御)

1. **鍵リスト管理**: プロジェクトメタデータで有効ユーザー鍵のホワイトリスト管理
2. **タイムスタンプベース**: 各ユーザーの最後の有効タイムスタンプを記録
3. **プロジェクト鍵即座更新**: 失効時に再生成・再配布

### 失効レベル

- **一時失効**: 指定期間のみアクセス停止
- **永続失効**: 完全削除
- **権限降格**: 編集 → 閲覧へ
- **部分失効**: 特定機能のみ制限

### 緊急時対応

- **不正アクセス検知**: 全ユーザー一時失効 + 再認証要求
- **デバイス紛失**: 該当デバイスローカル鍵無効化
- **退職・離脱**: 全プロジェクトから即座除名

## 7. プロジェクト単位権限管理

### 同期方式別の実装

| 方式 | 実装 |
| --- | --- |
| ローカル専用 | デバイス所有者が全権限。OS のファイル権限に依存。マルチユーザー時は追加暗号化 |
| クラウドストレージ同期 | プロジェクトディレクトリごとに独立暗号化。共有フォルダ招待 = プロジェクト鍵共有 |
| Web 同期 | サーバーサイドで細かい権限制御。API レベルで操作権限チェック。詳細監査ログ |

### 権限マトリックス

| 操作 | 所有者 | 編集者 | 閲覧者 | コメンター |
| --- | --- | --- | --- | --- |
| プロジェクト削除 | ✓ | - | - | - |
| メンバー招待・除名・権限変更 | ✓ | - | - | - |
| タスク作成・編集・削除 | ✓ | ✓ | - | - |
| タスク閲覧 | ✓ | ✓ | ✓ | ✓ |
| コメント投稿・ファイル添付 | ✓ | ✓ | - | ✓ |
| エクスポート | ✓ | ✓ | ✓ | - |

### セキュリティ境界

- **プロジェクト境界**: プロジェクト間データ完全分離
- **ユーザー境界**: アクセス権保持プロジェクトのみ参照可能
- **デバイス境界**: 紛失時も他デバイスに影響なし
- **時間境界**: 失効処理により過去鍵では新データアクセス不可

## 8. 今後実装すべきセキュリティ項目

### Phase 1 (高優先度)

- **脅威モデル分析**: 想定攻撃シナリオ (マルウェア / フィッシング / 内部不正 / 物理アクセス)、攻撃ベクター分析、リスク評価マトリックス
- **実装セキュリティ**: メモリ保護 (秘密鍵の即時クリア、ヒープダンプ対策)、サイドチャネル攻撃対策、本番デバッグ情報除去
- **監査・ログ**: 操作ログ詳細記録、認証失敗・不正アクセス試行の記録、ログ整合性検証

### Phase 2 (中優先度)

- **バックアップ・リカバリ**: 秘密分散法による鍵分割保存、災害復旧計画 (RTO/RPO)、マスター鍵紛失復旧手順
- **デバイス管理**: 信頼済みデバイス登録、証明書ベース認証、リモートワイプ
- **通信セキュリティ**: 証明書ピニング、TLS 1.3+ 強制、PFS、JWT + OAuth 2.0 / OIDC

### Phase 3 (低優先度)

- **高度な攻撃対策**: ゼロトラスト、AI ベース異常検知、TPM/HSM 活用
- **コンプライアンス**: GDPR (個人データ保護・削除権)、SOC 2 Type II、ISO 27001
- **パフォーマンス**: ハードウェア暗号化、セキュアな鍵キャッシュ、暗号化処理並列化

## 9. 実装ガイドライン

### 開発原則

1. **Secure by Default**: デフォルトで最も安全な設定
2. **Defense in Depth**: 多層防御
3. **Principle of Least Privilege**: 最小権限の徹底
4. **Fail Securely**: 障害時も安全な状態を維持
5. **Security Testing**: セキュリティテストの自動化

### コードレビュー観点

- 秘密情報のハードコーディング禁止
- 入力値検証の徹底
- SQL/NoSQL インジェクション対策
- XSS/CSRF 対策
- 暗号化実装の正確性

### 運用時の監視

- 異常な認証試行パターン
- 大量データアクセス
- 権限昇格試行
- システムリソース異常使用
- 暗号化処理の異常
